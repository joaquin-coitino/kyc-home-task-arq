KYC Process Analysis

Identifying Inefficiencies & Actionable Recommendations

Period: Jul 25, 2023 – Sep 22, 2023

Prepared for ARQ  ·  March 2026

1. Objective

Identify issues and inefficiencies in ARQ's KYC process (as of September 2023) and propose actionable improvements.

2. Findings & Recommendations

Each finding is paired with a specific recommendation. Findings are ordered by estimated business impact.
Finding 1 — Conversion Pipeline blockages account for 44% of all rejections

2,977 (44.0%) of rejections occurred because the Extraction check never ran, stalling all downstream document checks. These are not genuine verification failures — they are infrastructure or UX problems. Sub-causes: Extraction blocked despite Usability passing (2,022), Usability WARNING cascading downstream (530), and Usability not executing at all (425).

Recommendation Diagnose and fix pipeline blockages

Investigate vendor API timeouts, upload failures, and whether Usability WARNINGs should truly block Extraction. Implement retry logic and pre-submission image quality checks in-app. Recovering 50% of these cases would add ~1,488 passed users.

Finding 2 — Conversion Mexico's pass rate (81.5%) trails Argentina (92.1%) by 10.6 percentage points

Mexican users are disproportionately rejected. The gap is partly driven by MEX National ID having an 89.3% pass rate vs. Electoral ID at 95.8% — a 6.5pp difference within the same market. UNSUPPORTED_DOCUMENT_TYPE (509 usability WARNINGs) also suggests users are submitting unsupported document types that could be intercepted earlier.

Recommendation Guide Mexican users toward Electoral IDs and add a document type selector

Add in-app guidance recommending Electoral ID as the preferred document for Mexican users. Introduce a document type selector before the capture step to prevent unsupported documents from reaching the pipeline at all.

Finding 3 — Conversion 965 liveness rejections are non-conclusive

Of 1,032 liveness rejections, 965 are LIVENESS_UNDETERMINED — per Jumio docs this means the system could not reach a confident verdict, not that the user failed. These are almost always caused by poor lighting, glasses, or partial face visibility rather than spoofing attempts.

Recommendation Introduce a guided retry flow for LIVENESS_UNDETERMINED

Instead of hard-rejecting these users, prompt them to retry with specific guidance (better lighting, remove glasses, ensure full face is visible). This distinction between UNDETERMINED and hard fraud signals (e.g. ID_USED_AS_SELFIE) should drive different user flows.

Finding 4 — Operations Rejection rate nearly doubled during Aug 21 – Sep 3

The weekly rejection rate spiked from a baseline of ~10.7% to 18.7% during this two-week window, then partially recovered but remained above baseline through the end of the observation period. The root cause is not identifiable from the dataset alone.

Recommendation Investigate the root cause of the spike with Jumio

Cross-reference with Jumio SLA/incident reports, marketing campaign dates, and app release history. Hypotheses to test: vendor degradation, new user acquisition batch, increased fraud volume, or a UI change in the capture flow.

Finding 5 — Compliance 69 users matched a sanctions/PEP watchlist and all were passed

Per Jumio docs, a Watchlist WARNING carries the label ALERT, meaning the user was found on one or more global or regional sanctions lists or is a Politically Exposed Person. All 69 matches received an overall PASSED outcome. It is unclear from the data whether a manual review process exists for these cases.

Recommendation Establish a documented manual review process for watchlist ALERT matches

Confirm with the compliance team that each ALERT match is individually reviewed, that a documented decision rationale exists, and that the current pass-through policy is explicitly approved by compliance leadership.

Finding 6 — Compliance Manual overrides lack a documented policy

1 user was manually APPROVED despite similarity=NO_MATCH — their selfie did not match their ID. 2 users with liveness=REJECTED (LIVENESS_UNDETERMINED) received an overall PASSED. There is no evidence in the data of a formal override policy or audit trail for these decisions.

Recommendation Audit and formalise manual override decisions

Each manual override should have a documented rationale, an authorised approver, and be logged for audit purposes. If LIVENESS_UNDETERMINED is intentionally treated as non-blocking, this policy should be formalised in writing with compliance sign-off.

Finding 7 — Data Quality Multiple data quality issues indicate gaps in pipeline validation

Non-standard decision labels (OK, APPROVED, PASSES), inconsistent label casing (liveness_UNDETERMINED vs LIVENESS_UNDETERMINED), and implausible age values suggest the data pipeline lacks schema validation and output contracts with the vendor.

Recommendation Enforce data quality at the pipeline level

Add schema validation to reject non-standard label values in real time. Define and enforce a data contract with Jumio covering all expected field values, ensuring analytics remain reliable as the API evolves.

3. Methodology

This document and supporting analysis was primarily done on Python and HTML using Claude Code. The repository with the code and changes is available in Github.

The process to solve the problem was "greedy". We did not start with any hypothesis or predetermined workflow. We started by providing Claude Code with the description of the task and datasets and requested it to output a solution. We then recursively reviewed the output and, as soon as we identified the first material mistake (or suboptimal part), we requested Claude Code to fix and update the analysis (or parts of it). In order to ensure each new version Claude Code generated was better than the previous we provided the missing context (e.g.: Jumio docs, my own observations, summary of interview with a subject matter expert) - some of that context was stored in the repo itself to prevent regressions.

4. Limitations & Assumptions

The following limitations and open questions were identified during the analysis. They do not invalidate the findings but should be considered when acting on recommendations.

AreaLimitation / Assumption
Retry behaviourThe dataset contains one row per user with no retry history. It is unknown if some users re-attempted KYC and ultimately passed. Retry success rates would materially change the conversion impact estimates.
Workflow anomaliesFor 201 users, their usability check appears as not executed while at the same time the downstream checks have all passed. These may be users that have used a different workflow not shown on the dataset or, more likely, a data error.
Manual override policy4 users appear as APPROVED in the data. The exact difference between that label and PASSED is unclear — APPROVED may represent a manual decision and PASSED an automated one.
Liveness issue in usability check 278 users show a liveness related failure in the usability check which is not expected as per Jumio's API documentation. This could reflect a pipeline bug or API change. We have assumed for these cases that the failure occurred at the liveness check stage and not at the usability one.
API version changesThe dataset is from 2023; the Jumio documentation used reflects the current API (2026). Some label names or decision behaviours may have changed. This introduces a risk of the data being misinterpreted.
Time periodThe analysis covers ~2 months (Jul–Sep 2023). Seasonal patterns, long-term trends, and year-on-year comparisons are not possible with this data.
Dataset consistency1 user appears as PASSED in KYC_Summary but REJECTED in KYC_Details. The authoritative source for this record is unclear but we have decided to use KYC_Summary.
Colombia missingAccording to ARQ's website in July 2023, ARQ was live in Colombia (this can be seen using Wayback Machine). However, no records of Colombian users appear in the data set. At the time the company was known as DolarApp.

About the data

Two datasets analysed: KYC_Summary (one row per attempt with date and outcome) and KYC_Details (one row per attempt with the result of each individual check). Both contain 52,075 unique user references with no duplicates.

Date range: Jul 25, 2023 – Sep 22, 2023 (59 days)

Countries: Mexico (MEX) - 29,706 attempts  |  Argentina (ARG) - 22,369 attempts

Document types: ID Card (44,638), Passport (2,660), Driving License (972), Visa (24)

KYC checks performed: Usability, Extraction, Image Checks, Data Checks, Liveness, Similarity, Watchlist Screening.

Conclusions

  • Warnings are only ~1% of total attempts so we will be focusing on the larger problem of rejections.
  • Analysing both Mexico and Argentina is important as they have roughly similar volumes.
  • We will only look at ID cards and passports as they represent 98% of documents used by users.

5.3 Qualitative Review of Jumio

To complement the quantitative data, a conversation was held with Agustín Pividori, FinCrime lead at Personal Pay (a LATAM fintech and active Jumio customer). The insights below reflect his direct experience operating Jumio in production.

Key Points

  • Personal Pay selected Jumio primarily on cost.
  • FaceTec is the main competitor (widely used by Mercado Libre and major banks).
  • DNI (national ID) is the only universally required document type across LATAM financial institutions. Passports and driver's licenses are much less common (note: consistent with our dataset where ID Cards represent 86% of all attempts).
  • Jumio's document capture is aggressive in rejecting images that do not meet its quality bar. This creates friction.
  • False positives were a concern prior to SLA negotiations; once managed, they came within agreed limits.

Conclusions

  • Although unlikely to be visible in the current data set, we will keep an eye during the analysis on false positives that might merit a discussion with Jumio.
  • Verify if image quality rejections are excessive as this might indicate an issue with Jumio and warrant considering alternative vendors.

5.5 Pass Rates

Pass rates are analysed across three dimensions: time (weekly trend), document type, and country.

Trend over time

Rejection spike: Aug 21 – Sep 3 The rejection rate jumped from a baseline of ~10.7% to 16.1% (Aug 21–27) and 18.7% (Aug 28–Sep 3) — a 1.8× increase. The rate partially recovered but remained above baseline through the end of the observation period. Root cause is investigated in section 5.7.

By document type

Passports achieve the highest pass rate (96.4%) followed by Visas (95.8%). ID Cards (92.6%) and Driving Licenses (89.5%) perform below the overall average.

Despite the lower pass rate, ID Cards account for 44,638 attempts (86%) of all attempts — making them the most impactful document type to optimise.

Within ID Cards, Mexico's Electoral IDs (95.8%) outperform National IDs (89.3%) by 6.5 percentage points. Whether pipeline blockages explain the ID card gap is analysed in section 5.7.

By country

There is a 10.6 percentage-point gap in pass rates between Mexico (29,706 attempts, 57%) and Argentina (22,369 attempts, 43%).

Mexico's lower rate is partly driven by document mix — a higher share of National IDs (89.3%) vs Electoral IDs (95.8%). RESIDENT_PERMIT_ID in Mexico achieves 96.0%, showing the quality bar is reachable. Argentina's weakest performer is Driving Licenses (87.4%).

Whether pipeline blockages explain the MEX–ARG gap is analysed in section 5.7.

5.6 Rejection Causes

Rejections are attributed to the first check that blocked the pipeline, then drilled into the specific detail label returned by each check.

Root cause of rejections

Pipeline blockages — 2,977 rejections (44.0% of all rejections) These users were not rejected because a check explicitly failed — the pipeline itself stalled.
  • Extraction blocked (2,022): Usability passed but Extraction never ran, halting Image Checks, Data Checks, and Watchlist Screening.
  • Usability WARNING cascaded (530): A usability warning was treated as a hard blocker for Extraction.
  • Usability not executed (425): The root check itself did not run.
Similarity failures — 1,000 rejections (14.8%) Selfie does not match the document photo. Similarity runs independently of the document chain.
Liveness failures — 908 rejections (13.4%) System cannot confirm the user is live. Mostly UNDETERMINED (UX/connectivity) rather than hard spoofing signals.
Image check failures — 945 rejections (14.0%) Documents flagged as manipulated or digital copies.
Usability hard rejections — 933 rejections (13.8%) Document image could not be processed; blocks the entire document chain.

Failure reasons by check

Usability Failures

REJECTED: MISSING_MANDATORY_DATAPOINTS (407) and image quality issues (BLURRED, GLARE, MISSING_PAGE) — addressable with better in-app capture guidance. WARNING: UNSUPPORTED_DOCUMENT_TYPE (509) dominates — users submitting document types not supported by the workflow. Catchable earlier with a document selector.

Image Check Failures

MANIPULATED_DOCUMENT (537) is the dominant failure reason — genuine fraud signals that should be escalated for manual review. DIGITAL_COPY (225) may indicate users photographing a screen instead of a physical document.

Liveness Failures

LIVENESS_UNDETERMINED (965 cases) accounts for the vast majority. Per Jumio docs this means the system could not reach a confident verdict — almost always a UX or connectivity issue, not spoofing. A guided retry flow would recover most of these users.

Similarity Failures

NO_MATCH (1,104) — selfie and document photo could not be matched (hard rejection). NOT_POSSIBLE (0) — comparison cannot be determined (WARNING per docs). These 0 users pass overall without a confirmed face match — a potential compliance gap.

5.7 Anomaly Investigation

Three anomalies were identified in the pass rate analysis: a rejection spike in late August, a lower pass rate in Mexico compared to Argentina, and a lower pass rate of ID Cards compared to Passports. This section investigates whether pipeline blockages explain each one.

Q1 — Does the August rejection spike correlate with pipeline blockages?

Pipeline blockages partially explain the August spike During the spike weeks (Aug 21 – Sep 3), pipeline blockages accounted for 50.4% of rejections, compared to 42.1% during the baseline weeks — a notable 8.3pp shift. The increase in pipeline blockages during the spike suggests a vendor-side or infrastructure issue may have contributed to the elevated rejection rate.

Q2 — Do pipeline blockages explain Mexico's lower pass rate vs Argentina?

Pipeline blockages are a primary driver of the MEX–ARG gap In Mexico, 9.5% of total attempts end as pipeline blockages, vs 0.6% in Argentina — a 8.9pp difference. This gap is material and contributes meaningfully to the overall pass rate differential. Fixing MEX pipeline blockages would directly close part of the 10.6pp gap.

Q3 — Do pipeline blockages explain why ID Cards underperform Passports?

Pipeline blockages partially explain the ID Card–Passport gap For ID Cards, 0.6% of total attempts end as pipeline blockages, vs 0.7% for Passports — a -0.1pp difference. ID Cards are structurally more susceptible to pipeline blockages: they require front-and-back capture (increasing upload failure risk), have stricter field extraction requirements across many more regional variants, and are more likely to fail usability checks due to wear and regional printing differences. A share of the pass rate gap between ID Cards and Passports is therefore attributable to pipeline architecture, not document quality alone.
