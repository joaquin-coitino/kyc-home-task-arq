KYC Process Analysis

Identifying Inefficiencies & Actionable Recommendations

Analysis of 52,075 KYC attempts across two markets (MEX & ARG)
Period: Jul 25, 2023 – Sep 22, 2023

Prepared for ARQ  ·  March 2026

1. Objective

Identify issues and inefficiencies in ARQ's KYC process (as of September 2023) and propose actionable improvements.

2. Findings & Recommendations

Each finding is paired with a specific recommendation. Findings are ordered by estimated business impact.

Finding 1 — Conversion Pipeline blockages account for 44% of all rejections

2,977 (44.0%) of rejections occurred because the Extraction check never ran, stalling all downstream document checks. These are not genuine verification failures but rather infrastructure problems.

Recommendation Diagnose and fix pipeline blockages

Investigate vendor API timeouts, upload failures, and whether Usability WARNINGs for unsupported docs should truly block Extraction. Implement retry logic and pre-submission image quality checks in-app.

Finding 2 — Conversion Some document types are dealing to more failures

MEX National ID have an 89.3% pass rate vs. Electoral ID at 95.8%. 7.5% of failures are attributed to users uploading an unsupported document. Document photo issues such as glare, blurr, data is hard to read, etc. represent over 10% of failures.

Recommendation Guide users during document upload

Add in-app guidance recommending Electoral ID as the preferred document for Mexican users. Make it clear for users what documents are not supported. Provide better guidance on how to take a photo of the document.

Finding 3 — Conversion 965 liveness rejections are non-conclusive

Of 1,032 liveness rejections, 965 are LIVENESS_UNDETERMINED. Per Jumio docs this means the system could not reach a confident verdict, not that the user failed. These are mostly caused by poor lighting, glasses, movements not done as expected, or partial face visibility rather than spoofing attempts.

Recommendation Introduce a guided retry flow for LIVENESS_UNDETERMINED

Instead of hard-rejecting these users, prompt them to retry with specific guidance (better lighting, remove glasses, move head in certain direction, ensure full face is visible). This distinction between UNDETERMINED and hard fraud signals (e.g. ID_USED_AS_SELFIE) should drive different user flows. This might require a discussion with Jumio or considering a different vendor.

Finding 4 — Operations Rejection rate nearly doubled during Aug 21 – Sep 3

The weekly rejection rate spiked from a baseline of ~10.7% to 18.7% during this two-week window, then partially recovered but remained above baseline through the end of the observation period.

Recommendation Investigate the root cause of the spike with Jumio

Cross-reference with Jumio SLA/incident reports, marketing campaign dates, and app release history. Hypotheses to test: vendor degradation, new user acquisition batch, increased fraud volume, or a UI change in the capture flow.

Finding 5 — Data Quality Multiple data quality issues indicate gaps in pipeline validation

Typos, inconsistent casing, misrouting of data to the wrong column, etc. suggest the data pipeline lacks schema validation and output contracts with the vendor and internal services.

Recommendation Enforce data quality at the pipeline level

Add schema validation to reject non-standard label values in real time. Define and enforce a data contract with Jumio and internal services covering all expected field values, ensuring analytics remain reliable as the API evolves.

Finding 6 — Compliance Passed underage users

There are 1,286 users with a year of birth recorded that would imply they were less than 18 years old in 2023. 95% of them passed KYC.

Recommendation Review policy on minor onboarding.

If this is a data quality issue, then records need to be updated. Otherwise, ARQ needs to have a clear stance on onboarding minors in compliance with applicable law.

3. Methodology

This document and supporting analysis was primarily done on Python and HTML using Claude Code. The repository with the code and changes is available in Github.

The process to solve the problem was "greedy". We did not start with any hypothesis or predetermined workflow. We started by providing Claude Code with the description of the task and datasets and requested it to output a solution. We then recursively reviewed the output and, as soon as we identified the first material mistake (or suboptimal part), we requested Claude Code to fix and update the analysis (or parts of it). In order to ensure each new version Claude Code generated was better than the previous we provided the missing context (e.g.: Jumio docs, my own observations, summary of interview with a subject matter expert). To prevent regressions, some context ia stored in the repo itself so Claude Code can refer to it and each change is reviewed before committing and pushing.

4. Limitations

The following limitations were identified during the analysis. They do not invalidate the findings but should be considered when acting on recommendations.

AreaLimitations
Unclear retry behaviourThe dataset contains one row per user with no retry history. It is unknown if some users re-attempted KYC and ultimately passed. Retry success rates would materially change the conversion impact estimates.
Data quality issuesMultiple instances have been found of data that seems incorrect. However, we are unable to confirm if they are actually errors and what the correct value should be.
API version changesThe dataset is from 2023; the Jumio documentation used reflects the current API (2026). Some label names or decision behaviours may have changed. This introduces a risk of the data being misinterpreted.
Short time periodThe analysis covers ~2 months (Jul–Sep 2023). Seasonal patterns, long-term trends, and year-on-year comparisons are not possible with this data.
No Colombian usersAccording to ARQ's website in July 2023, ARQ (DolarApp) was live in Colombia (this can be seen using Wayback Machine). However, it is not known why there are no records of Colombian users in the data set.
Narrow scope of dataThe dataset only gives a small view of ARQ's KYC flows. No access has been provided to logs, Jumio support, codebase, etc.. However, those elements are key to diagnosing and solving the problem.

About the data

Two datasets analysed: KYC_Summary (one row per attempt with date and outcome) and KYC_Details (one row per attempt with the result of each individual check). Both contain 52,075 unique user references with no duplicates.

Date range: Jul 25, 2023 – Sep 22, 2023 (59 days)

Countries: Mexico (MEX) - 29,706 attempts  |  Argentina (ARG) - 22,369 attempts

Document types: ID Card (44,638), Passport (2,660), Driving License (972), Visa (24)

KYC checks performed: Usability, Extraction, Image Checks, Data Checks, Liveness, Similarity, Watchlist Screening.

Priorities for Analysis
  • Warnings are only ~1% of total attempts so we will be focusing on the larger problem of rejections.
  • Analysing both Mexico and Argentina is important as they have roughly similar volumes.
  • We will only look at ID cards and passports as they represent 98% of documents used by users.

5.2 Jumio Documentation

Jumio's documentation explains the different checks that are part of the KYC pipeline.

Check Overview

CheckWhat it testsPossible decisions
Usability Whether uploaded images (ID and selfie) are of sufficient quality to process PASSED, REJECTED, WARNING, NOT_EXECUTED
Extraction Whether mandatory fields can be extracted from the ID PASSED, NOT_EXECUTED
Image Checks Whether the ID passes integrity tests PASSED, REJECTED, WARNING, NOT_EXECUTED

Liveness Whether the selfie was captured from a live person (detects spoofing, printed photos, screen recordings) PASSED, REJECTED, WARNING, NOT_EXECUTED
Similarity Whether the face in the selfie matches the face on the ID document PASSED (MATCH), REJECTED (NO_MATCH), WARNING (NOT_POSSIBLE), NOT_EXECUTED

5.3 Qualitative Review of Jumio

To complement the quantitative data, a conversation was held with Agustín Pividori, FinCrime lead at Personal Pay (a LATAM fintech and active Jumio customer). The insights below reflect his direct experience operating Jumio in production.

Key Points

  • Personal Pay selected Jumio primarily on cost.
  • FaceTec is the main competitor (widely used by Mercado Libre and major banks).
  • National ID is the only universally required document type across LATAM financial institutions. Passports and driver's licenses are much less common (note: consistent with our dataset where ID Cards represent 86% of all attempts).
  • Jumio's document capture is aggressive in rejecting images that do not meet its quality bar. This creates friction.
  • False positives were a concern prior to SLA negotiations; once managed, they came within agreed limits.
Things to Keep an Eye On
  • Although unlikely to be visible in the current data set, we will keep an eye during the analysis on false positives that might merit a discussion with Jumio.
  • Verify if image quality rejections are excessive as this might indicate an issue with Jumio and warrant considering alternative vendors.

5.4 Data Cleaning

Several data quality issues were identified in the dataset. They can affect downstream analytics and thus we had to make a decision on how to correct the data.

Liveness value misrouted to usability278 rows have liveness_UNDETERMINED in usability_decision_details and in liveness_decision_details. The value from the latter seems to have been copied by mistake to the former.We have assumed the usability check was passed and the failure occurred in the liveness check.
PASSED with usability not executed201 users have usability_decision=NOT_EXECUTED with detail NOT_UPLOADED, yet all downstream checks (Extraction, Image Checks, Liveness, Similarity) passed.We have assumed the usability check was passed instead of being not executed.
Non-standard top-level decision labels12 rows use OK (8) or APPROVED (4) in decision_label instead of PASSED; 1 row has PASSED in usability_decision_details instead of OK.We have assumed OK, APPROVED and PASSED to be equivalent to each other.

Fixing Data Quality Issues The first 3 issues seem the result of pipeline problems that should be fixed at the source.

5.5 Pass Rates

Pass rates are analysed across four dimensions: time (weekly trend), country, document type, age.

Trend Over Time

The rejection rate jumped from a baseline of ~10.7% to 16.1% (Aug 21–27) and 18.7% (Aug 28–Sep 3), i.e.: a 1.8× increase. The rate partially recovered but remained above baseline through the end of the observation period.

By Country

There is a 10.6 percentage-point gap in pass rates between Mexico and Argentina.

By Document Type

Passports have the higher pass rates (96.4%) compared to ID Cards (92.6%). This is understandable as passports are more standardised.

Within ID Cards, Mexico's Electoral IDs (95.8%) outperform National IDs (89.3%) by 6.5 percentage points.

ID Cards only as they are by far the most common document type (44,638 attempts, 86% of all attempts).

By Age Group

Among adult cohorts, pass rates are broadly stable, ranging from 91.3% (65+) to 93.8% (18–24), which is a spread of only 2.5 percentage points.

Unknown (3,789 users, 0.5% pass rate): No year_birth recorded. It is concerning there are any passes without a year of birth.

5.6 Rejection Causes

Rejections are attributed to the first check that blocked the pipeline.

Pipeline blockages: 2,977 rejections (44.0% of all rejections) These users were not rejected because a check explicitly failed but instead because the pipeline itself stalled.
  • Extraction blocked (2,022): Usability passed but Extraction never ran, halting Image Checks, Data Checks, and Watchlist Screening.
  • Usability WARNING cascaded (530): A usability warning was treated as a hard blocker for Extraction because document was unsupported.
  • Usability not executed (425): The root check itself did not run. Could be that the user did not provide the document or that there is a bug preventing processing.

Usability Failures

Usability failures are partially solvable with better UX.

Image Check Failures

Manipulation failures are likely fraud. It is a good thing they have failed. Digital copy failures could be legitimate and some of them can be addressed with better UX.

Liveness Failures

Liveness undetermined failures are mostly due to UX.

Similarity Failures

No match failures are likely actual cases of fraud. Face matching tech is very mature so mistakes are rare.

5.7 Anomaly Investigation

We will review the impact of pipeline blockages vs hard rejections on the rejection spike in late August and a lower pass rate in Mexico compared to Argentina.

August Rejection Spike

Pipeline blockages partially explain the August spike During the spike weeks (Aug 21 – Sep 3), pipeline blockages accounted for 50.4% of rejections, compared to 42.1% during the baseline weeks, a notable 8.3pp shift. The increase in pipeline blockages during the spike suggests a vendor-side or infrastructure issue may have contributed to the elevated rejection rate.

Mexico vs Argentina Pass Rate

Pipeline blockages are a primary driver of the MEX–ARG gap In Mexico, 9.5% of total attempts end as pipeline blockages, vs 0.6% in Argentina which is a 8.9pp difference. This gap is material and contributes meaningfully to the overall pass rate differential. Fixing MEX pipeline blockages would directly close part of the 10.6pp gap.
